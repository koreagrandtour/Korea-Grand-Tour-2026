# Cross-Check Rules

Use the smallest check that fits the risk. Do not reread the full project unless the source of truth is unclear.

## Token Rule

Default to:

1. newest user instruction
2. relevant context file
3. source file / live tool readback
4. full handoff only when blocked

Keep check notes compact: `OK / Fix / Blocked + evidence`.

## Perspectives

For important, public, sponsor-facing, participant-facing, legal/safety, website, GitHub, or high-risk output, review from:

- project manager
- marketing / PR
- operations
- safety / legal
- privacy
- KGT brand / tone
- factual accuracy

For low-risk internal work, use only the relevant perspectives.

## Checklist

- Confirm dates, prices, deadlines, route status, sponsor status, and participant counts.
- Keep 2025 actual metrics separate from 2026 projections.
- Mark missing information as `OPEN QUESTION`.
- Do not expose private participant data.
- Do not reveal full route details before planned reveal timing.
- Do not present contacted sponsors as confirmed.
- Do not call Korea Grand Tour a race.
- Make safety rules direct and unambiguous.
- Keep tone warm, human, direct, premium but not corporate.
- Confirm Korean copy sounds natural if Korean is used.
- Confirm output matches the requested channel and audience.

## Required Contract

- Producer self-checks before handoff.
- Receiver cross-checks inherited facts before reuse.
- Corrected output is re-checked.
- High-risk work records evidence in `.agents/workflow_status.md` when a task has multiple stages.

## Red Flags

Stop and ask François if:

- a file may expose private participant/payment/insurance data
- sponsor claims are unclear
- legal wording could create liability
- route information may be unreleased
- a public post includes private documents, private conversations, unrevealed route details, or otherwise sensitive context
- the source of truth is unclear

## Output Note

For high-risk work, include a short note that the cross-check was done and list any remaining risks or open questions.
